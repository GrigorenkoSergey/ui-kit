## Что это за репа?
Здесь была попытка создать простенькое SPA приложение, удовлетворяющее следующим требованиям:
* Использования сторонних библиотек сокращено до минимума. Используем только сборщик (выбран webpack) и нативные возможности браузеров. Да и то, сборщик (как сторонняя технология) выбран исключительно для возможности дальнейшего подключения TS.
По факту можно было бы обойтись вообще без него, может быть было бы проще. Уж слишком много ньюансов возникает с использованием сборщиков. 
* Cтраницы и стили должны подгружаться лениво (только по мере необходимости).

По факту получился свой мини-фреймворк или starter-kit.

Соотвествующие статьи на Хабр: [1](https://habr.com/ru/articles/953566/), [2](https://habr.com/ru/articles/958026/), [3](https://habr.com/ru/articles/962778/).

## Роутинг и менеджер состояния
Здесь создан свой собственный роутинг и менеджер состояний. 
Код каждой страницы должен храниться в директории `pages`.
Внутренние ссылки помечаем атрибутом `data-inner-link`.

Все, что должно храниться в памяти приложения, и быть доступным для всех страниц, храним в директории `stores` в отдельных файлах.

## SEO
На каждой странице включено только то, что ей необходимо и в полном объеме. Сложные элементы будут генерироваться с помощью [web-components](https://developer.mozilla.org/en-US/docs/Web/API/Web_components), которые более-менее дружат с поисковыми роботами. Таким образом, SEO получаем из коробки. 

## Разработка
Уникальный код каждой страницы находится в соответствующей директории `pages` в отдельной папке. В каждой такой папке должны быть файлы `index.html`, `index.js`, `style.css`. Можно создавать глубоко вложенные страницы, они будут также найдены и обработаны.

```
src/pages/
├── page-1
│   ├── index.html
│   ├── index.js
│   ├── page-1-deeper-page
│   │   ├── index.html
│   │   ├── index.js
│   │   └── style.css
│   └── style.css
├── page-2
│   ├── index.html
│   ├── index.js
│   └── style.css
└── page-404
    ├── index.html
    ├── index.js
    └── style.css
```
Каждый `index.js` будет являться начальной точкой сборки страницы. В экспорте по умолчанию должна быть повторяющаяся логика страницы. Достаточно один раз зайти на страницу и логика будет сохранена в памяти приложения. При каждом заходе на страницу, функция, содержащая логику, будет запущена заново. 
Данная функция может возвращать функцию очистки памяти (к примеру, очистки подписок DOM-элементов на хранилища). Эта функция будет запущена при переходе по страницам. 

```js
// src/pages/some-page-name/index.js
const logic = () => {
  ...
  const cleanup = derive(() => { ... });

  return cleanup;
};

export default logic;
```

Во время сборки `html`-код страницы помещается в [шаблон](./src/page-template.html) (см. правило `test: /(index|404)\.html$/i`). Обратите внимание на относительные пути в шаблоне и тег `base`.

Далее вставляется общий код `main.js` в каждой странице (см. настройку `HtmlWebpackPlugin`). Там содержатся общий скрипт для всех страниц. В частности подключение роутинга и общие стили. Когда пользователь заходит на любую существующую страницу, скрипт применяется, и далее переходы по страницам будут в рамках текущей сессии, без полной перезагрузки страницы.

Замечу, что в папке `.storybook` должен находиться свой `tsconfig.json`, иначе пути неверно определяются. Причина неизвестна, возможно в точке перед storybook.
